说起来有点不好意思——搭个个人博客这件事，我从两年前就说要做了。

那时候我想的是"等我有空了就搞"，结果你也知道，「有空」这件事永远不存在。直到上个月某个周末，我决定不等了。

## 为什么又重新折腾博客

之前的博客用的是某平台自带的，写起来是方便，但总觉得不是自己的东西。就像租房子和买房子的区别——平台说改版就改版，说下线就下线，你连个备份都找不到。

而且，平台博客有个致命问题：**你写的东西不属于你，属于平台。**

SEO、自定义域名、代码高亮、暗色模式……这些在平台上要么没有，要么要花钱。但对我来说，这些都是刚需。毕竟来读技术博客的人，大概率也是开发者，他们对体验是有要求的。

## 技术选型：为什么选了 Astro

老实说，我一开始想用 Next.js，毕竟全栈框架，什么都能做。但转念一想——我需要的是博客，不是 web app。

对比了几轮：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Next.js** | 全栈、生态好 | 配置复杂、bundle 大 |
| **Hugo** | 极快、Go 模板 | 学习曲线陡 |
| **Hexo** | 简单、插件多 | 性能一般 |
| **Astro** | 零 JS 默认、Markdown 原生 | 新，社区相对小 |

最后选了 Astro，核心原因就一个：**它默认不往页面塞 JS**。

一个博客，读者来是为了看文字和代码，不是为了执行 500KB 的 JavaScript。Astro 的架构正好契合这个场景——静态生成，按需 hydration，加载速度直接起飞。

## 踩过的坑

### 坑 1：中文路径和 slug

中文文章标题自动生成 slug 的时候，有些框架会出问题。比如"我是怎么搭起这个博客的"可能变成一串乱码。

Astro 默认用的 slugify 处理得还行，但有些特殊字符需要手动处理。我的解决方法很简单：文章文件名直接用英文，中文标题放在 frontmatter 里。

```bash
# 文件名用英文
src/data/blog/how-i-built-this-blog.md
```

### 坑 2：代码高亮的主题

默认的代码高亮主题在暗色模式下不太好看。后来换成了 Shiki 的 `night-owl` 主题（暗色）和 `min-light` 主题（亮色），配合 Astro 的 data-theme 属性自动切换。

配置在 `astro.config.ts` 里：

```ts
shikiConfig: {
  themes: { light: "min-light", dark: "night-owl" },
  defaultColor: false,
},
```

### 坑 3：字体加载

中文网页最大的挑战是字体。思源黑体（Noto Sans SC）文件很大，全量加载会让首屏变慢。

解法：
1. 用 Google Fonts 的 CDN，支持按需加载
2. 设置好 fallback 字体栈：`Noto Sans SC, Inter, system-ui`
3. 用 `font-display: swap` 避免文字闪烁

## 最终效果

现在这个博客：
- ✅ 支持暗色模式
- ✅ 代码高亮好看
- ✅ 移动端适配
- ✅ SEO 友好
- ✅ RSS 订阅
- ✅ 零 JS 加载（除了暗色模式切换）

部署在 Vercel 上，每次 push 到 main 分支自动构建。写文章就是写 Markdown，`git push` 就上线，简单粗暴。

## 给想搭博客的人的建议

1. **别等"准备好了"再开始** — 我等了两年，其实一周就够了
2. **别过度设计** — 先上线，再迭代。内容比设计重要
3. **选你写起来舒服的工具** — 如果写文章本身就很痛苦，你不会坚持写的
